# Release Check Skill

1つまたは複数のPRに対して、リリース予定のJIRAチケットの内容がすべて含まれているかを検証する。

## 使用方法

### パターン1: 単一PR URLのみ（チケット番号を自動抽出）

```
/release-check {PR URL}
```

PR本文からチケット番号（`BAX-XXXX`）を自動抽出してチェックする。

### パターン2: 単一PR URL + チケット番号を明示指定

```
/release-check {PR URL} BAX-1234 BAX-5678 ...
```

### パターン3: 複数PR URL（チケット番号を自動抽出）

```
/release-check {PR URL1} {PR URL2} {PR URL3} ...
```

複数PRの変更内容を統合してチェックする。各PRからチケット番号を自動抽出し、重複を排除して統合する。

### パターン4: 複数PR URL + チケット番号を明示指定

```
/release-check {PR URL1} {PR URL2} BAX-1234 BAX-5678 ...
```

引数:
- **PR URL**: GitHub PRのURL（1つ以上必須、複数指定可）
- **チケット番号**: `BAX-XXXX` 形式（省略可、スペース区切りで複数指定可）

引数が不足している場合はAskUserQuestionで確認する。

### 引数の判別ルール

- `https://github.com/` で始まる引数はPR URLとして扱う
- `BAX-` で始まる引数はチケット番号として扱う

## 実行手順

### Step 1: PR情報の取得

**各PRについて並列で**以下を実行する:

```bash
gh pr view {PR URL} --json title,body,files,commits
```

PRから以下を収集:
- **タイトル**: PRのタイトル
- **本文**: PRの説明文
- **変更ファイル一覧**: 変更されたファイルのリスト
- **コミット一覧**: コミットメッセージのリスト

さらに、PRの差分の概要を把握する:
```bash
gh pr diff {PR URL} --name-only
```

#### 複数PRの場合の統合

複数PRが指定された場合、以下を統合して1つのデータセットとして扱う:
- **変更ファイル一覧**: 全PRの変更ファイルをマージ（重複排除）
- **コミット一覧**: 全PRのコミットメッセージを結合
- **本文**: 各PRの本文をすべて参照対象とする

各PRのタイトル・URLは個別に記録し、レポートで表示する。

### Step 2: チケット番号の確定

#### チケット番号が引数で渡された場合

そのチケット番号をそのまま使用する。

#### チケット番号が渡されていない場合（PR URLのみ）

**全PRの本文**からチケット番号を自動抽出する。以下のパターンで検索:

1. **JIRAチケットURL**: `https://kokopelli-inc.atlassian.net/browse/BAX-XXXX` から `BAX-XXXX` を抽出
2. **チケット番号の直接記載**: `BAX-XXXX` パターンにマッチする文字列を抽出
3. **コミットメッセージ**: コミットメッセージ内の `BAX-XXXX` パターンも補助的に参照

複数PRの場合、全PRから抽出したチケット番号を統合し、重複を排除する。

抽出したチケット番号の一覧をユーザーに提示し、確認を求める:
- 「PR本文から以下のチケットを検出しました。これでチェックしてよいですか？」
- チケット番号のリストを表示
- ユーザーが確認したら続行、追加・削除があれば反映

**チケット番号が1つも見つからない場合**: AskUserQuestionでチケット番号を直接聞く。

### Step 3: JIRAチケット情報の取得（並列実行）

各チケットについて `mcp__plugin_atlassian_atlassian__getJiraIssue` を使用:

- **cloudId**: `kokopelli-inc.atlassian.net`
- **issueIdOrKey**: `BAX-XXXX`

各チケットから以下を収集:
- **タイトル（Summary）**: チケットの概要
- **説明（Description）**: チケットの詳細説明
- **ステータス**: チケットの状態
- **サブタスク**: 子チケットがあればそのリスト

### Step 3.5: マイグレーションファイルの検出

PRの変更ファイル一覧から、マイグレーションファイルの有無をチェックする。

#### 検出対象パターン

以下のパスパターンにマッチするファイルをマイグレーションファイルとして検出:

- `**/migrations/**`
- `**/migrate/**`
- `**/*migration*`
- `**/*migrate*`
- `**/schema/**`
- `**/db/**/*.sql`
- `**/database/**/*.sql`
- `**/alter_*.sql`
- `**/create_*.sql`

#### 検出時の処理

マイグレーションファイルが1つでも検出された場合、以下の情報を収集:
- ファイルパス一覧
- 変更内容の概要（ALTER TABLE, CREATE TABLE, DROP等のDDL操作）

この情報はStep 5の結果レポートで強調表示する。

### Step 4: マッチング分析

各チケットについて、PRに含まれているかを以下の基準で判定:

#### チェック項目

1. **チケット番号の言及**: PRタイトル、本文、コミットメッセージにチケット番号（BAX-XXXX）が含まれているか
2. **内容の関連性**: チケットの説明内容とPRの変更ファイル・差分が関連しているか
3. **コミットの紐付け**: チケット番号を含むコミットが存在するか

#### 判定ロジック

各チケットを以下のいずれかに分類:

| 判定 | 条件 |
|------|------|
| **含まれている** | チケット番号がコミットメッセージまたはPR本文に言及されている |
| **おそらく含まれている** | チケット番号の直接言及はないが、変更内容がチケットの説明と関連している |
| **含まれていない** | チケットに関連する変更が見つからない |
| **判定不能** | チケットの説明が不十分で判断できない |

### Step 5: 結果レポート

以下のフォーマットで結果を表示:

```markdown
## リリースチェック結果

**チェック日時**: {YYYY-MM-DD HH:MM}

### 対象PR

| # | PR | タイトル |
|---|-----|---------|
| 1 | {PR URL1} | {PRタイトル1} |
| 2 | {PR URL2} | {PRタイトル2} |

※ 単一PRの場合は `**PR**: {PR URL}` のシンプル表示でもよい。

### マイグレーション警告

> **🚨 このPRにはマイグレーションファイルが含まれています！**
>
> リリース時にDBマイグレーションの実行が必要です。デプロイ手順を確認してください。
>
> | ファイル | 操作内容 |
> |---------|---------|
> | `path/to/migration_file.sql` | ALTER TABLE xxx ADD COLUMN ... |
> | `path/to/another_migration.php` | CREATE TABLE xxx ... |

※ マイグレーションファイルが検出されなかった場合、このセクションは省略する。

### サマリー

- 対象チケット数: {N}
- 含まれている: {N}
- おそらく含まれている: {N}
- 含まれていない: {N}
- 判定不能: {N}

### 詳細

#### ✅ 含まれている

| チケット | タイトル | 根拠 |
|---------|---------|------|
| BAX-XXXX | {タイトル} | コミット `abc1234` で言及 |

#### ⚠️ おそらく含まれている

| チケット | タイトル | 根拠 |
|---------|---------|------|
| BAX-YYYY | {タイトル} | `src/xxx.php` の変更が関連 |

#### ❌ 含まれていない

| チケット | タイトル | 備考 |
|---------|---------|------|
| BAX-ZZZZ | {タイトル} | 関連する変更が見つかりません |

#### ❓ 判定不能

| チケット | タイトル | 理由 |
|---------|---------|------|
| BAX-WWWW | {タイトル} | チケットの説明が不十分 |

### 注意事項

{必要に応じて補足コメント}
```

### Step 6: PRコメントへの投稿

結果レポートをPRのコメントとして投稿する。

#### 投稿前の確認

結果をユーザーに表示した後、AskUserQuestionで確認する:
- 「この結果をPRコメントとして投稿しますか？」

#### 投稿方法

**各PRに対して**結果レポートをコメントとして投稿する（複数PRの場合は全PRにコメント）:

```bash
gh pr comment {PR URL} --body "$(cat <<'EOF'
{結果レポートのMarkdown}

---
*このコメントはClaude Codeのrelease-checkスキルにより自動生成されました*
EOF
)"
```

#### 既存コメントの更新

同じPRに既にrelease-checkのコメントが存在する場合（フッターの文言で判定）、新しいコメントを追加する（上書きはしない）。

### Step 7: 確認とフォローアップ

- 「含まれていない」チケットがある場合、ユーザーに確認:
  - 「以下のチケットがPRに含まれていないようです。対応が必要ですか？」
- 「おそらく含まれている」チケットがある場合:
  - 「以下のチケットは関連する変更がありますが、チケット番号の明示的な言及がありません。確認してください。」

## エラーハンドリング

| 状況 | 対応 |
|------|------|
| PR URLが無効 | エラーメッセージを表示 |
| `gh` CLI未認証 | 「`gh auth login` を実行してください」と案内 |
| JIRAチケットが見つからない | 該当チケットを「判定不能」に分類し、理由を記載 |
| JIRAにアクセスできない | WebFetchでチケットページを取得してフォールバック |

## 注意事項

- PRの差分が大きい場合は、ファイル名とコミットメッセージを中心に判定する
- チケットのサブタスクがある場合は、サブタスクも含めてチェックする
- 判定は保守的に行う（確信がない場合は「おそらく」または「判定不能」とする）
